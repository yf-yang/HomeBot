# HomeBot 软件架构设计与开发规划

本文档基于《技术方案选型.md》中的讨论内容，
为 HomeBot 家用机器人项目制定整体软件架构并形成详细的开发规划。

---

## 1. 目标 & 范围

- 构建轻量级、模块化、可扩展的机器人控制软件。
- 使用 ZeroMQ 为核心通信机制，保留后续与 MQTT 的混合可能。
- 先搭建框架，不写业务实现；后续按照规划逐步填充。
- 适配树莓派/Jetson 等资源有限硬件。

---

## 2. 总体架构概览

采用**分层+模块化**设计，按职责划分为三个主要层次：

1. **应用层 (Applications)**
   - 手机遥控 (Web/移动端)
   - 语音交互 (LLM 接口)
   - 模仿学习 (动作捕捉/记录)
   - 人体跟随 (CV 算法)
2. **服务层 (Services)**
   - 运动控制服务 (MotionService)
   - 视觉处理服务 (VisionService)
   - 语音处理服务 (SpeechService)
3. **硬件抽象层 (HAL)**
   - 底盘驱动 (ChassisDriver)
   - 机械臂驱动 (ArmDriver)
   - 相机驱动 (CameraDriver)
   - 音频驱动 (AudioDriver)

各层通过 ZeroMQ 端点互连，消息协议采用 JSON 定义类型与数据。

架构图已在《技术方案选型.md》中描述，此处以文字再现。

## 3. 技术栈与依赖

- **语言**：Python 3.11（主控），C/C++ 可用于性能敏感的 HAL 驱动
- **通信**：pyzmq (ZeroMQ Python 绑定)
- **消息格式**：JSON（考虑未来 Protobuf/MsgPack 升级）
- **AI/ML**：PyTorch 或 TensorFlow，用于视觉与语音模型
- **Web 前端**：React/Vue + WebSocket → 转 ZMQ
- **部署工具**：Dockerfile, Makefile, shell 脚本
- **测试**：pytest, unittest
- **CI/CD**：GitHub Actions（代码检查、构建、单元测试）

## 5. 模块说明

### 5.1 Common
- 消息类型、日志封装、配置管理、错误码。

### 5.2 应用层
每个子模块通过 ZeroMQ PUB/SUB 与对应服务交换数据。

### 5.3 服务层
- **MotionService**：接收速度/位置控制命令，调用 HAL 驱动。
- **VisionService**：订阅摄像头数据，发布检测结果。
- **SpeechService**：处理语音输入/输出，可能调用云端 LLM。

### 5.4 硬件抽象层
统一接口暴露给上层，内部负责与底层驱动或操作系统交互。

## 7. 开发阶段划分

### 阶段 1：项目初始化（1-2 周）
1. 搭建仓库结构与 CI 配置
2. 实现基础框架代码：common、ZeroMQ 抽象层
3. HAL 驱动骨架（空接口）
4. 编写简单示例服务和客户端
5. 文档编制：架构图、开发规范

### 阶段 2：核心功能开发（3-4 月）
1. MotionService 与底盘/机械臂驱动联调
2. VisionService 基于摄像头传输图像、简单的障碍检测
3. SpeechService 集成本地语音识别与 TTS
4. 开发基本应用模块：遥控、简单跟随
5. 持续编写单元测试与集成测试

### 阶段 3：高级功能与优化（3-6 月）
1. 人体跟随高级算法、模仿学习
2. 路径规划、避障策略
3. 系统性能优化：延迟、资源占用
4. 引入 MQTT 桥接、云端功能
5. 机器人 OTA 升级机制

### 阶段 4：部署与发布
1. 制作 Docker 镜像/固件包
2. 用户手册、维护文档
3. Beta 测试与反馈
4. 正式发布

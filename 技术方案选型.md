## 1. 软件架构方案（无 ROS）

### 1.1 架构设计

采用**分层模块化 + ZeroMQ 消息总线**架构：


### 1.2 核心通信机制：ZeroMQ

| 模式 | 用途 | 示例 |
|------|------|------|
| **PUB-SUB** | 传感器数据广播 | 相机图像、底盘里程计 |
| **REQ-REP** | 同步控制命令 | 机械臂移动到指定位置 |
| **PUSH-PULL** | 任务队列 | 视觉处理任务分发 |
| **PAIR** | 双向实时控制 | 手机虚拟摇杆 → 底盘速度 |

### 1.4 不用 ROS 的理由

- HomeBot 是轻量级家庭机器人，不需要 ROS 的复杂生态
- 树莓派/Jetson 资源有限，ROS2 开销过大（~1GB vs ZeroMQ ~1MB）
- 核心功能（遥控、跟随、抓取）用 ZeroMQ 完全够用
- 启动时间：ZeroMQ <1秒 vs ROS2 5-10秒

## 2. MQTT vs ZeroMQ 对比

| 维度 | MQTT | ZeroMQ |
|------|------|--------|
| **架构** | 星型（Broker 中心节点） | 网状（点对点，无中心） |
| **部署** | 需运行 Broker（Mosquitto/Eclipse） | 无需 Broker，直接 socket 连接 |
| **代码量** | 客户端简单，但需管理 Broker | 稍复杂，但无外部依赖 |
| **延迟** | 2-5ms（本地）+ Broker 处理 | <1ms（直接通信） |
| **可靠性** | QoS 0/1/2 可选，持久化消息 | 需自行实现，默认无持久化 |
| **实时性** | 中等，Broker 可能成为瓶颈 | 高，无中间节点 |
| **跨网络** | 天然支持（NAT穿透、TLS） | 需自行处理网络拓扑 |
| **资源占用** | Broker 占用内存 | 纯客户端，无额外进程 |
| **调试难度** | 可用 mosquitto_sub/pub 直接调试 | 需写代码或抓包 |

### 2.1 HomeBot 场景分析

| 场景 | 要求 | 评估 |
|------|------|------|
| 手机遥控 → 底盘 | 低延迟、高频率（20-50Hz）| ZeroMQ 更优 |
| 相机 → 视觉处理 | 高带宽、本地传输 | ZeroMQ 更优 |
| 语音交互 → LLM | 低频、需联网 | MQTT 也可以 |
| 多设备扩展（未来）| 跨局域网、云端控制 | MQTT 更优 |

### 2.2 推荐方案：混合架构

```
┌─────────────────────────────────────────┐
│              局域网内部（ZeroMQ）          │
│  ┌─────────┐    ┌─────────┐    ┌───────┐ │
│  │ 手机遥控 │←──→│ 机器人主控│←──→│ 机械臂 │ │
│  │ (WebSocket│   │ (ZeroMQ) │    │       │ │
│  │  转ZMQ) │    └────┬────┘    └───────┘ │
│  └─────────┘         │                    │
│                      ↓                    │
│              ┌─────────────┐              │
│              │ 视觉/语音服务 │              │
│              │  (PUB-SUB)  │              │
│              └─────────────┘              │
└─────────────────────────────────────────┘
                      │
                      │ MQTT（可选桥接）
                      ↓
┌─────────────────────────────────────────┐
│           云端/远程控制（MQTT）            │
│         用于：OTA升级、远程监控、多设备管理    │
└─────────────────────────────────────────┘
```

**结论**：先用 ZeroMQ 把核心功能跑起来，如果以后需要手机远程控制（不在同一WiFi），再在网络边界加一个 MQTT 桥接节点。

## 3. 单摄像头视觉避障方案

### 3.1 方案对比

| 方案 | 原理 | 计算量 | 精度 | 适用场景 |
|------|------|--------|------|----------|
| **光流法 (Optical Flow)** | 像素运动估计深度 | 低 | 中 | 快速移动、紧急避障 |
| **单目深度估计 (MiDaS/Fast-Depth)** | CNN 估计深度图 | 中 | 中高 | 通用避障、路径规划 |
| **尺寸扩张法 (Size Expansion)** | 物体在图像中变大=靠近 | 极低 | 低 | 正前方障碍物检测 |
| **端到端分类 (Blocked/Free)** | 直接分类是否可通行 | 低 | 中 | 简单环境、快速原型 |
| **语义分割+地面检测** | 分割可通行区域 | 中高 | 高 | 复杂室内环境 |

### 3.2 推荐方案

#### 方案1：端到端二分类（最快实现）

**原理**：训练一个轻量 CNN，直接判断"前方是否被阻挡"

```
输入：单帧图像 (224x224)
输出：Blocked (被阻挡) / Free (可通行) 概率
```

**实现**：
```python
import torch
import torchvision.models as models
from PIL import Image

# 使用 MobileNetV3 (轻量级)
model = models.mobilenet_v3_small(num_classes=2)
model.load_state_dict(torch.load('obstacle_model.pth'))
model.eval()

# 推理
def is_blocked(frame):
    img = preprocess(frame)  # resize to 224x224
    with torch.no_grad():
        output = model(img)
        prob_blocked = torch.softmax(output, dim=1)[0][0].item()
    return prob_blocked > 0.5  # 阈值可调
```

**避障逻辑**：
```python
while True:
    frame = camera.capture()
    if is_blocked(frame):
        robot.stop()
        robot.turn_left()  # 或根据历史帧选择方向
    else:
        robot.forward()
```

**优点**：简单、速度快（Jetson Nano 30+ FPS）、不需要深度估计
**缺点**：只能判断"能否前进"，不能规划路径

#### 方案2：单目深度估计（推荐）

**原理**：用轻量 CNN 从单帧图像估计深度图

**推荐模型**：

| 模型 | 参数量 | Jetson Nano FPS | 特点 |
|------|--------|-----------------|------|
| **Fast-Depth** | ~3M | 20-30 | 专为嵌入式优化 |
| **MiDaS-small** | ~14M | 10-15 | 精度更高 |
| **PyD-Net** | ~1M | 30+ | 最快，精度够用 |

**避障逻辑**：
```python
import cv2
import torch

depth_model = torch.hub.load("intel-isl/MiDaS", "MiDaS_small")

def avoid_obstacle(frame):
    # 1. 估计深度图
    depth = depth_model(frame)
    
    # 2. 将图像分成左/中/右三个区域
    h, w = depth.shape
    left_depth = depth[:, :w//3].mean()
    center_depth = depth[:, w//3:2*w//3].mean()
    right_depth = depth[:, 2*w//3:].mean()
    
    # 3. 选择最远的方向
    depths = {'left': left_depth, 'center': center_depth, 'right': right_depth}
    best_direction = max(depths, key=depths.get)
    
    if center_depth < 0.5:  # 前方0.5米内有障碍
        if best_direction == 'left':
            robot.turn_left()
        else:
            robot.turn_right()
    else:
        robot.forward()
```

#### 方案3：光流法（适合动态场景）

**原理**：利用连续帧的像素运动估计场景深度

**核心思想**：
- 近处物体在图像中移动快（光流大）
- 远处物体在图像中移动慢（光流小）

**实现**：
```python
import cv2

def optical_flow_depth(prev_frame, curr_frame):
    # 计算稠密光流
    flow = cv2.calcOpticalFlowFarneback(
        prev_gray, curr_gray, None,
        pyr_scale=0.5, levels=3, winsize=15,
        iterations=3, poly_n=5, poly_sigma=1.2, flags=0
    )
    
    # 光流幅度 = sqrt(u^2 + v^2)
    flow_magnitude = np.sqrt(flow[..., 0]**2 + flow[..., 1]**2)
    
    # 光流大的区域 = 近处障碍物
    return flow_magnitude

# 避障：检测光流大的区域占比
def check_obstacle(flow_magnitude, threshold=0.5):
    center_region = flow_magnitude[h//3:2*h//3, w//3:2*w//3]
    return center_region.mean() > threshold
```

**优点**：不需要神经网络、计算量极低
**缺点**：需要机器人移动才能估计深度、对旋转敏感

### 3.3 HomeBot 具体建议

考虑到硬件（树莓派/Jetson Nano + 单摄像头）和需求：

| 阶段 | 方案 | 理由 |
|------|------|------|
| **快速验证** | 方案1 端到端分类 | 1小时训练，立即见效 |
| **日常避障** | 方案2 MiDaS-small | 平衡精度和速度 |
| **极限优化** | 方案3 光流法 | 无神经网络依赖 |

**避障策略伪代码**：
```python
class VisualAvoidance:
    def __init__(self):
        self.depth_model = load_midas_small()
        self.safe_distance = 0.5  # 米
        
    def decide_movement(self, frame):

        # 分区域分析
        regions = self.split_regions(depth_map)
        
        if regions['center'].min() < self.safe_distance:
```
